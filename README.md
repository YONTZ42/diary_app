📜 API更新・破壊的変更の運用ルール
当プロジェクトはフロントとバックが分離しているため、以下の Expand/Contract（拡張と収縮）戦略 を徹底します。

① フィールドの変更・削除（破壊的変更）が必要な場合
Expand（拡張）: バックエンドに 「新しいフィールド」を追加 し、旧フィールドは残したままマージ・リリースする。

Migrate（移行）: フロントエンド側で、新しいフィールドを参照するようにコードを修正・リリースする。

Contract（収縮）: 全てのフロントエンド（本番環境含む）が旧フィールドを参照しなくなったことを確認後、バックエンドから 旧フィールドを削除 する。

② CIによるガード
Backend CI: oasdiff により、意図しない破壊的変更が含まれるPRをブロックします。意図的な場合は、その旨をレビューで明記し、例外的にマージします。

Frontend CI: バックエンドの最新スキーマをフェッチして型チェックを行い、バックエンドの変更に追従できていないコードをデプロイさせません。
